# 多端一致性与测试策略

本项目最终会同时有 Windows 电脑版、Android 手机版和 Web 版。它们面向同一批任务数据，但代码和界面并不相同：

- Windows 版目前使用 Python + Tkinter。
- Android 版目前使用 Flutter。
- Web 版计划使用 React + Vite + TypeScript。
- 同一个“添加任务”“完成任务”“提醒任务”，在两端可能各有一份实现。

因此，多端软件很容易出现一种问题：电脑版已经调试过很多次，但手机版可能还藏着同类 bug；或者手机版修好了，电脑版却没有同步修复。

## 核心风险

多端不一致通常来自这些地方：

- 两端字段不一致，例如电脑有 `reminder`，手机忘了保存。
- 两端校验规则不一致，例如电脑不允许空标题，手机却允许。
- 两端提醒逻辑不一致，例如电脑能到点提醒，手机只保存了提醒开关。
- 两端网络同步逻辑不一致，例如手机写入云端，电脑还只读本地文件。
- 两端测试覆盖不一致，例如电脑测过“过去时间不能提醒”，手机没测。

## 解决思路

### 1. 把核心规则写清楚

不能只靠记忆维护规则。项目需要明确规定：

- 任务标题不能为空。
- 有提醒时必须有日期和时间。
- 过去时间不应该创建新的提醒。
- 删除任务必须从云端删除。
- 已完成任务不应该继续提醒。
- 每个用户只能看到自己的任务。

这些规则后续应该同时体现在文档、代码和自动化测试里。

### 2. 做自动化测试

自动化测试的作用是：每次改代码后，让程序自动帮我们检查关键功能有没有坏。

后续电脑端和手机端都应该逐步补测试，例如：

- 空标题不能添加。
- 添加任务后能重新读取。
- 完成状态能保存。
- 删除后不会再显示。
- 日期时间能保存。
- 提醒开关能保存。
- 只选择日期、不选择时间时不能创建有效提醒。

这样可以减少“肉眼测试漏掉的问题”。

### 3. 尽量共用后端规则

重要的数据规则应该尽量放在 Supabase 这类后端服务里，而不是只放在某一个客户端里。

例如：

- 用户只能访问自己的任务。
- 任务必须属于某个用户。
- 任务标题不能为空。
- 任务字段结构统一。

这样 Windows 和 Android 都必须遵守同一套底层规则。

### 4. 做功能对照表

后续需要维护一张 Windows / Android / Web 功能一致性表：

| 功能 | Windows | Android | Web | 状态 |
|---|---|---|---|---|
| 添加任务 | 完成 | 完成 | 计划中 | Web 待做 |
| 删除任务 | 完成 | 完成 | 计划中 | Web 待做 |
| 完成/未完成 | 完成 | 完成 | 计划中 | Web 待做 |
| 日期时间 | 完成 | 完成 | 计划中 | Web 待做 |
| 到点提醒 | 完成 | 未完成 | 暂不做 | 不一致 |
| 搜索 | 完成 | 未完成 | 暂不做 | 不一致 |
| 筛选 | 完成 | 未完成 | 暂不做 | 不一致 |
| 云同步 | 完成 | 完成 | 计划中 | Web 待做 |

这样项目状态不会依赖记忆，而是可以一眼看出哪里还没对齐。

### 5. 做真机测试和场景测试

手机端尤其需要真机测试，因为它会受到系统权限、后台限制、网络状态等影响。

后续需要测试：

- App 在前台时提醒是否正常。
- App 在后台时提醒是否正常。
- 小米手机是否限制后台通知。
- 网络断开后再恢复，任务是否还能同步。
- 电脑和手机同时修改同一个任务时会怎样。
- 同一个账号在两台设备登录时，数据是否一致。

## 本项目建议执行顺序

1. 先维护“功能一致性表”。
2. 给 Android 端补关键自动化测试。
3. 让 Web 端接入 Supabase。
4. 对齐 Windows、Android、Web 的任务字段、校验规则、同步规则。
5. 做 Windows、Android、Web 三端同步测试。
6. 再考虑 v3.0 正式发布。

目标不是让两个端完全长得一样，而是让它们遵守同一套规则、处理同一批数据、在关键功能上表现一致。
