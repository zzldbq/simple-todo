# Simple Todo v2.0 设计

## 目标

v2.0 让 Windows 与 Android 使用同一账号，并同步同一批任务。

## 系统结构

```text
Windows 客户端（Python） ─┐
                         ├─ Supabase Auth + PostgreSQL
Android 客户端（Flutter）─┘
```

两个客户端不直接互相连接。它们都登录 Supabase，并从云数据库读取和修改任务。

## 账号方案

- 第一版只支持邮箱和密码注册、登录、退出。
- 每个账号由 Supabase 生成唯一用户 ID。
- 客户端只保存登录会话，不保存明文密码。
- 暂不加入手机号、微信或第三方登录。

## 云端任务模型

每条任务包含：

- `id`：任务唯一 ID。
- `user_id`：任务所属用户。
- `title`：任务内容。
- `due_at`：日期和时间，统一存为带时区的时间。
- `reminder`：是否提醒。
- `completed`：是否完成。
- `notified`：当前设备是否已处理提醒；后续可能改为设备级提醒记录。
- `created_at`：创建时间。
- `updated_at`：最后修改时间。

## 安全规则

数据库开启 Row Level Security（RLS）：

- 用户只能查看自己的任务。
- 用户只能创建属于自己的任务。
- 用户只能修改和删除自己的任务。
- 客户端只能使用可公开的 publishable/anon key，绝不包含 service-role key。

## 同步规则

第一阶段采用简单可靠的方案：

1. 登录后从云端拉取全部任务。
2. 添加、编辑、完成或删除时，先提交云端。
3. 云端成功后刷新本地界面。
4. 打开应用或手动刷新时重新拉取。
5. `updated_at` 较新的记录视为最新版本。

实时推送、离线冲突合并和后台同步放到后续迭代，避免第一版同步逻辑过于复杂。

## 本地数据迁移

Windows 版首次登录时：

1. 检查本地 `tasks.json`。
2. 询问用户是否上传本地任务。
3. 上传成功后保留本地备份，避免数据丢失。
4. 后续以云端数据为主。

## Android 提醒

- 云同步只负责保存任务和提醒时间。
- Android 客户端使用本机通知实现提醒。
- 通知权限、时区和重启后的提醒恢复需要单独测试。

## 暂不包含

- iPhone 客户端。
- 多人共享任务。
- 微信登录。
- 文件附件。
- 跨设备即时推送提醒。
- 复杂离线冲突合并。
